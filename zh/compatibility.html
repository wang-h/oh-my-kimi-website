兼容性边界

Kimi-native 优先，受 oh-my-codex 启发，但不是盲目照搬。

oh-my-kimi 借鉴的是“oh-my-* 增强层”这种思路，但它最终要服务的是 Kimi CLI。 所以有些概念会继承，有些会改写，有些则明确不做。

Kimi-native 部分

  • 主包名：oh-my-kimi
  • 主命令名：omk
  • 公开文档与网站品牌
  • 围绕 Kimi CLI 重写后的 shell 定位

继承来的思路

  • “工作流外壳”这个核心模型来自 oh-my-codex
  • setup + skills + team + state 是主要组成
  • 对上游来源保持明确引用，而不是假装没有历史
  • 迁移决策写清楚，而不是藏起来

这个页面最想说明什么

你应该把 oh-my-kimi 理解成“给 Kimi CLI 用的增强层”，而不是一个泛多智能体站点，也不是一个没有说明来源的静默 rename。 这就是兼容性页面最重要的边界说明。
