Built for Kimi CLI · OMK

这是给 Kimi CLI 用的 oh-my-codex 风格增强层。

oh-my-kimi 给 Kimi CLI 补上更完整的外壳：setup、skills、可复用工作流、team 执行、运行时辅助和本地持久化状态。 如果你见过 oh-my-codex 或 oh-my-claudecode，这就是同类思路——只是目标明确对准 Kimi CLI。

快速开始 查看 GitHub 浏览文档
Kimi CLI 仍然是执行引擎；oh-my-kimi 提供的是工作流外壳与运行时增强。
在 Kimi CLI 里，建议优先使用 /skill:ralplan、/skill:deep-interview、/skill:team、/skill:ralph 这类显式调用。$name 是 OMK 风格简写，不应默认假设 Kimi 原生支持。
一个更蓝、更明确、面向 Kimi 的公开品牌页面。

这个项目到底是什么

执行引擎

Kimi CLI

真正负责 agent 执行、会话和 CLI 交互的，仍然是 Kimi CLI 本身。

增强层

oh-my-kimi

在上面补上 setup、skills、文档、team、状态目录和更明确的工作流外壳。

同类参考

oh-my-codex / oh-my-claudecode

思路相似，但 oh-my-kimi 的目标是 Kimi CLI，而不是其他 CLI。

为什么会有人用它

更稳定的工作流外壳

不再每次都临时拼提示词，而是沿着一套固定的规划—执行—验证流程走。

Kimi-first 的 setup

安装后能直接进入更完整的 Kimi CLI 使用方式，而不是只多一个命令名。

Team + runtime 能力

在 Kimi CLI 上叠加 tmux team、HUD、MCP、通知和本地状态，不用自己从零搭。

本地持久化项目记忆

计划、日志、状态和流程产物都留在本地，适合跨会话持续推进。

快速开始

  1. npm install -g oh-my-kimi
  2. omk setup
  3. omk doctor
  4. omk

几分钟内就能把 Kimi CLI 变成一个更完整的工作流外壳。

核心工作流

进入 Kimi CLI 之后，oh-my-kimi 希望你按更稳定的方式工作。

/skill:deep-interview 澄清任务
/skill:ralplan 批准实施计划
/skill:ralph 持续推进到完成
/skill:team 3:executor "把已批准的工作并行化"
